# 06 — UI: Integrations Tab

> React components for Settings → Integrations. Mockup: `onboarding.html` in this folder. Uses the canonical SIGNAL design language from `docs/design/design_language.html`.

---

## Scope

Three surfaces:

1. **Integrations list** (`/settings/integrations`) — connected integrations with state dots + row actions.
2. **Add wizard** — provider picker + auth field form + verifying + success.
3. **Reconnect flow** — same wizard, entering at the auth field form, scoped to an existing integration.

Existing Settings tab hosts it. No first-run integration with the setup wizard in v1 (integrations are an ongoing activity, not a first-boot task).

## Data flow

```
useIntegrations()
  ├── initial GET /api/integrations
  └── SSE subscription to /api/integrations/events
       └── state changes push updates into React state

useCatalog()
  └── GET /api/integrations/catalog (cached in memory for session)

useAddIntegration()
  ├── POST /api/integrations
  ├── returns {id, state, state_reason} on completion
  └── verifying state is inferred from response latency
     (add-endpoint blocks until supervisor returns terminal state)
```

## IntegrationsList

Matches the mockup's "populated list" view. Each row:

- Icon (from catalog entry).
- Label (user-editable inline).
- State badge: `active` / `auth_failed` / `error` / `pending` / `disabled`.
- Permission badge: `Read only` / `Read and write`.
- Secondary text: account identifier (e.g. email), last verify time.
- Row actions: **Verify**, **Reconnect** (shown only for `auth_failed`), **Disable/Enable toggle**, **Delete**.
- Edit drawer (pencil icon) carries the full permissions radio — same component as Step 2 of the wizard. Saving fires `PATCH /api/integrations/:id { write_allowed }`. The supervisor respawns the broker with the new flag (~1–3 s); the SSE channel pushes `active → pending → active` and the row's state badge animates through the transition. No reconnection of the user's credentials — the same app password is re-used. UI feedback on toggle: save button shows a brief spinner, the row's status dot briefly shows the pending colour, then settles back to green. The write-enabled tool set in the agent's registry refreshes at the same moment.

Uses `.fs-page` + `.fs-row` design-language classes.

## AddIntegrationModal

Uses `.dl-modal` for the frame. Renders `WizardFrame` inside with a state machine:

```
  provider_picker → auth_fields → verifying → success
                                       │
                                       └─fail→ back to auth_fields
                                                with error banner
```

### Step 1 — Provider picker

- Grid of catalog entries grouped by category (`email_calendar`, `dev_tools`, `smart_home`, `productivity`, etc. — category field on catalog entry).
- "Custom MCP server" escape hatch always at the bottom.
- On click, loads `GET /api/integrations/catalog/:slug` (returns merged catalog + auth FIELDS) and advances.

### Step 2 — Auth fields (dynamic)

- Renders one form control per `Field` from the auth plugin:
  - `type: password` → `<input type="password">` with visibility toggle
  - `type: email` → `<input type="email">`
  - `type: token` → `<input type="password">` styled with monospace font
  - `type: url` → `<input type="url">`
  - `type: text` → `<input>`
- If `deep_link` present, render a `.btn-outline` button above the relevant field linking to the provider's docs page (opens in new tab).
- Plus three integration-level controls:
  - **Label** (default: `catalog.label_default`)
  - **Reference** = `<slug>_` prefix (non-editable) + suffix input (editable, default from label)
  - **Permissions** — two `.radio-card` options, *Read only* (default, selected) and *Read and write*. Copy is scoped to the catalog kind (e.g. email/calendar text for IMAP+CalDAV, repo-action text for GitHub). Submit payload includes `write_allowed: boolean`.
- Client-side validation enforces `pattern`, `min_length`, `max_length`. Pattern mismatches show inline hint.

### Step 3 — Verifying

Shows the progress checklist mockup. Since the POST is synchronous from the UI's POV (blocks until terminal state), the "checks" are illustrative:

- "Vault write" — done as soon as POST is in flight (~instant).
- "Broker spawn" — done at ~200ms mark.
- "Upstream auth" — the actual bottleneck; shown as spinner until response.

If we want real progress, supervisor could stream progress events over the SSE channel. Defer — the fake checklist is fine for v1.

### Step 4 — Success (or failure)

- Success: `.wz-body` with check icon, integration ID, summary table (services enabled, broker UID). "Done" and "Add another" buttons.
- Failure (auth_failed): inline error banner + re-enter the auth form, focusing the first field (usually `password`/`token`).
- Failure (error): banner + Retry button; Retry issues POST `/api/integrations/:id/verify`.

## Reconnect flow

Triggered from an existing integration row (state = `auth_failed` or from a "Change credentials" row action). Opens the same wizard UI, skipping the provider picker and using the existing `integration_id` + `label`. Form pre-filled only with non-secret fields (email). Secret fields are blank — user must re-enter.

Endpoint: `POST /api/integrations/:id/reconnect`.

## Error-state UX

| Situation | Treatment |
|---|---|
| Catalog request fails | Full-page error in Integrations tab with Retry; other tabs unaffected |
| Add POST returns 400 (validation) | Highlight offending fields, surface server error text |
| Add POST returns 409 (auth_failed) | Return to Step 2 with banner; fields preserved except secrets |
| Add POST returns 500 | Generic "Something went wrong" + Retry |
| SSE connection drops | Silently reconnect; UI state stays fresh from the last full GET |

## Component-local open items

- **First-run prompt.** The Settings screen exists but users may not know integrations are there. Option: after successful first-run (model pick), show a one-time banner on the main chat view: "Connect your email, calendar, and dev tools." Defer.
- **Inline tool test.** After successful add, offer "Try it out → Search email" button that runs a safe no-op tool call. Nice UX but not v1.
- **Custom MCP warning modal.** Per the mockup, pasting a `uvx/npx` command is an arbitrary-code-execution surface. Show a confirmation modal explaining the risk before first Custom MCP add.
- **`SettingsPage` should own its tab routing.** Today `DesktopApp.jsx` reads `settingsTab` and renders one of `<ProfilesTab/>`, `<IntegrationsTab/>`, `<SystemSettings/>` as `SettingsPage` children. `SettingsPage` already owns the tab bar, so it should also own the mapping from tab id → component, with per-tab props passed via a small prop bag. Removes the leak of "which tab maps to which component" out into `DesktopApp` and lets us add a 4th tab without touching the desktop shell.
- **Migrate `GoalDetailPanel` to the shared `ConfirmButton` primitive.** `components/primitives/ConfirmButton.jsx` (added with the integrations DELETE flow) is the canonical click-twice-to-confirm widget; `GoalDetailPanel.jsx` still has its own inline `confirmDelete` state + button. Swap to the primitive so the two delete flows share behavior (timeout, busy state, danger palette) and we have one place to fix bugs.
