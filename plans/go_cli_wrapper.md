# Go CLI Wrapper — `computron`

## What It Is

A single binary that replaces `docker run ...` with `computron start`. It handles Docker flags, GPU detection, volume setup, env var management, log rotation, version management, and health checks. Users install one binary — Docker and Ollama are the only prerequisites.

## CLI, Not GUI

A CLI is the right choice:

- **Target audience** — self-hosted AI users who are already comfortable with terminals (they're running Docker and Ollama)
- **Cross-platform** — a CLI binary works identically on Linux, macOS, and Windows (WSL2). A GUI means GTK/Cocoa/Win32 or Electron, tripling the build complexity for no real gain.
- **Composability** — users can script it, put it in cron, wrap it in systemd
- **Distribution** — single static binary, no runtime dependencies. A GUI needs frameworks.
- **Maintenance** — one codebase, not three platform-specific UIs

The web UI at `:8080` is the GUI. The CLI just manages the container lifecycle.

## Commands

### `computron start`

The main command. Progressive behavior:

1. Check for Docker — error with install link if not found
2. Check for Ollama — warn if not running, don't block (cloud models work without it)
3. Pull image if needed — `ghcr.io/lefoulkrod/computron_9000:latest`, show progress
4. Detect GPU — `nvidia-smi` check. If found:
   - Verify nvidia-container-toolkit (`nvidia-ctk --version`)
   - If missing, print setup instructions and offer to continue without GPU
   - If present, add `--gpus all`
5. Build `docker run` command from config + detected capabilities:
   - Always: `--shm-size=256m`, `--network=host`, `--log-opt max-size=50m --log-opt max-file=3`, named volumes
   - If GPU: `--gpus all`
   - Env vars from config: `HF_TOKEN`, `ENABLE_*` flags, `TELEGRAM_*`, etc.
6. Handle name conflicts — remove stopped container with same name
7. Wait for ready — poll `http://localhost:<port>/api/features` until it responds (with timeout)
8. Open browser — `xdg-open` / `open` / `start` depending on OS

```
Flags:
  --no-gpu         Skip GPU detection
  --no-open        Don't open browser after start
  --port N         Override web UI port (default 8080)
  --detach=false   Tail logs after start instead of backgrounding
  --version TAG    Use a specific image tag instead of latest
```

### `computron stop`

Stop and remove the container. Volumes persist.

### `computron restart`

Stop + start.

### `computron update`

Pull latest image. If container is running, prompt to restart (or `--yes` to auto-restart).

### `computron logs`

Tail container logs. `--follow` to stream.

### `computron status`

Show:
- Running / stopped
- Image version (tag + digest)
- Uptime
- GPU detected (yes/no, which card)
- Ollama reachable (yes/no)
- Port
- Feature flags enabled
- Volume sizes

### `computron config`

Persistent config so users don't pass flags every time:

```bash
computron config set hf_token hf_xxx
computron config set enable_image_gen true
computron config set port 9090
computron config list
computron config get hf_token
computron config reset
```

Stored in `~/.config/computron/config.yaml` (Linux), `~/Library/Application Support/computron/config.yaml` (macOS), `%APPDATA%\computron\config.yaml` (Windows).

```yaml
# Auto-generated by computron config
port: 8080
gpu: auto          # auto | always | never

# Tokens
hf_token: ""
github_token: ""
github_user: ""
telegram_bot_token: ""
telegram_chat_id: ""

# Features
enable_image_gen: false
enable_music_gen: false
enable_desktop: false
enable_grounding: false
enable_custom_tools: false

# Container
image: ghcr.io/lefoulkrod/computron_9000:latest
```

### `computron shell`

Open an interactive shell inside the running container.

### `computron reset`

Remove volumes and container. Fresh start. Requires `--yes` or interactive confirmation.

### `computron enable` / `computron disable` (sugar)

```bash
computron enable image_gen music_gen
computron disable desktop
```

Shorthand for `computron config set enable_image_gen true` etc. Restarts the container if running.

## Docker Interaction

Shell out to the Docker CLI (`docker run`, `docker stop`, etc.) — not the Docker Go SDK. Simpler, fewer dependencies, and the user already has the CLI installed. Parse JSON output with `docker inspect --format`.

## Building

```bash
cd cli
go build -o computron ./cmd/computron
```

Cross-compile:
```bash
GOOS=linux GOARCH=amd64 go build -o computron-linux-amd64 ./cmd/computron
GOOS=darwin GOARCH=arm64 go build -o computron-darwin-arm64 ./cmd/computron
GOOS=windows GOARCH=amd64 go build -o computron-windows-amd64.exe ./cmd/computron
```

## Publishing & Distribution

### GitHub Releases (primary)

Use [GoReleaser](https://goreleaser.com/) via GitHub Actions. On every tagged push (`v1.0.0`):

1. GoReleaser builds binaries for all platforms:
   - `computron-linux-amd64`
   - `computron-linux-arm64`
   - `computron-darwin-amd64`
   - `computron-darwin-arm64`
   - `computron-windows-amd64.exe`
2. Creates a GitHub Release with:
   - Binaries attached as assets
   - Auto-generated changelog from commits
   - SHA256 checksums file

`.goreleaser.yaml`:
```yaml
builds:
  - main: ./cmd/computron
    binary: computron
    env: [CGO_ENABLED=0]
    goos: [linux, darwin, windows]
    goarch: [amd64, arm64]
    ignore:
      - goos: windows
        goarch: arm64

archives:
  - format: tar.gz
    format_overrides:
      - goos: windows
        format: zip

checksum:
  name_template: checksums.txt

changelog:
  sort: asc
```

GitHub Actions workflow (`.github/workflows/release.yml`):
```yaml
on:
  push:
    tags: ['v*']
jobs:
  release:
    runs-on: ubuntu-latest
    permissions:
      contents: write
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-go@v5
        with: { go-version: '1.22' }
      - uses: goreleaser/goreleaser-action@v6
        with:
          args: release --clean
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

### Install Script (primary for Linux/macOS)

Hosted at a stable URL. Detects OS/arch, downloads the right binary from GitHub Releases:

```bash
curl -fsSL https://raw.githubusercontent.com/lefoulkrod/computron_9000/main/install.sh | sh
```

The script:
1. Detects `uname -s` / `uname -m`
2. Fetches the latest release tag from GitHub API
3. Downloads the matching binary
4. Installs to `~/.local/bin/` (or `/usr/local/bin/` if run with sudo)
5. Verifies SHA256 checksum
6. Prints next steps: `computron start`

### Windows

Windows users download `computron-windows-amd64.exe` from the GitHub Releases page and put it in their PATH. The install script doesn't target Windows — WSL2 users use the Linux binary inside WSL.

### Package Managers (later, not v1)

- **Homebrew tap** (macOS/Linux): `brew install lefoulkrod/tap/computron`
- **AUR** (Arch): community-maintained
- **Scoop** (Windows): `scoop install computron`

These are nice-to-haves but not worth the maintenance for v1.

## Implementation Order

1. **Skeleton** — cobra CLI, `start` / `stop` / `logs` / `status` subcommands, config file load/save
2. **Start command** — Docker detection, image pull, build run command, health wait, browser open
3. **GPU detection** — nvidia-smi + nvidia-ctk checks, guided setup messages
4. **Config** — persistent config, `enable`/`disable` sugar, env var passthrough
5. **Polish** — `update`, `reset`, `shell`, checksums, error messages
6. **Release** — GoReleaser, GitHub Actions, install script
7. **README** — rewrite install instructions around the CLI

## Open Questions

- **Auto-update**: Should `computron start` check for CLI updates? Adds latency but keeps users current. Maybe opt-in with `computron config set auto_update true`.
- **Ollama model management**: Should `computron start` auto-pull the vision model if missing? It's required for the desktop agent. Maybe prompt: "Vision model not found. Pull now? (3 GB) [Y/n]"
- **Telemetry**: Anonymous usage stats (which features are enabled, GPU vs CPU) would help prioritize development. But needs careful opt-in UX to not be creepy.
