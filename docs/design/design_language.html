4. Borders & Radius
Standard border weight is 1px. Heavier borders are reserved for the Signal Line (2px accent indicator on active elements — see Part II). The radius scale ranges from sharp pixel-aligned (--radius-sm) to pill-shaped (--radius-full). Match radius to component scale: small for badges, medium for buttons/inputs, large for cards and modals.

5. Shadows
Shadows are minimal. They appear only on floating elements (modals, flyouts, toasts) and interactive hover states. Resting cards and panels use borders, not shadows. The --shadow-glow is reserved for focus rings on inputs and primary buttons.

Part III
Building Blocks
The atomic components everything else is assembled from. Each one has a single responsibility and a defined visual treatment. Larger views (in Part IV) compose these primitives — the goal is that designing a new screen is a question of "which building blocks do I need" rather than "what does this look like."
8. Form Controls
Buttons, inputs, and selectables. All form controls run on a 32px standard height. Buttons sit on --radius-md; inputs and textareas use the same. The focus ring is always --shadow-glow.
Buttons
Icon buttons (28×28)
// When to reach for which. The four variants form a loudness ladder: filled > outline > ghost, with danger as a parallel destructive track.
// .btn-filled — the single primary action on a surface (Save, Connect, Submit, Next). At most one per row.
// .btn-outline — the default for everything that isn't the primary CTA. Cancel, Back, Retry, Configure. A real, peer choice the user might pick.
// .btn-ghost — tertiary / quiet. Use when the button stands alone or genuinely shouldn't compete for attention (a solo Skip, a hover-revealed row action). Reads as "an escape hatch."
// .btn-danger — destructive (Delete, Disconnect, Remove). Use anywhere the action can't be cheaply undone.
// Rule of thumb: if two buttons live in the same action row, neither should be ghost. Use filled for the one primary CTA, outline for the rest, danger for destructive. Reach for ghost only when the button is alone or you specifically want it to feel almost-not-there.
// Counter-example: a wizard footer of [Back] [Cancel] [Save]. All three are peer choices — render Back & Cancel as outline and Save as filled. A ghost Cancel sitting next to outline Back reads as disabled, especially in dark mode.
→ click 1 arms; click 2 fires
// Click-twice-to-confirm pattern. Destructive actions that can't be cheaply undone use a two-stage confirm: resting renders as btn-danger (danger-muted background, danger text); click 1 escalates to a solid danger fill (background: var(--danger); color: #fff) and the label flips to "Confirm?"; click 2 within the timeout (default 3s) fires the action.
// The armed-state escalation is the affordance — resting state alone is not loud enough to signal "your next click is destructive." Don't replace this with a modal for inline row/panel actions; reserve modals for multi-field destructive forms.
// Implemented by the ConfirmButton primitive (see server/ui/src/components/primitives/ConfirmButton.jsx). All Delete actions in the app should route through it.

Part IV
Composed Patterns
Multi-part assemblies that combine building blocks into reusable structures. Each pattern defines a shell (positioning, sticky regions, scroll behavior) plus how primitives sit inside it. Building a new screen is a question of "which patterns do I assemble" — Header on top, Sidebar on the left, Split Panel for master-detail, Tabbed Preview for parallel previews, etc.
12. Header
The top app bar, always visible. Logo + brand title on the left; ghost icon buttons on the right (audio, theme, settings, new conversation). Sits on --surface with a 1px border-bottom. Height: --header-height (36px).

13. Sidebar
Vertical icon navigation, always visible on desktop. Width: --sidebar-width (44px). Icons stack vertically; a separator divides primary destinations from secondary ones; the bottom reserves space for Settings. Active state shows the Signal Line on the left edge.
Layout: 28×28 icon buttons, --radius-md, gap: --sp-1 between buttons, padding: --sp-2 0 for the column. Separator is 20px wide, 1px solid --border, vertical margin --sp-1.

Active state: --accent-muted background, --accent color. Pseudo-element ::before draws a 2px accent bar on the left edge — the Signal Line.

README.md

SIGNAL Design Language

The design language for COMPUTRON 9000 — clear communication between human and machine.

Components reference semantic tokens (--canvas, --surface, --accent) rather than raw color primitives.

  • Signal over Noise
  • Instrument Panel
  • Computational Warmth
  • Depth through Layering
  • Monospace Moments
Spacing: toolbar padding --sp-2 --sp-3 on --surface. File pill in font-code at 12px with --accent-muted background. Toggle is two pill buttons inside a --canvas rounded container; active button gets --surface background. Content padding varies by renderer: 16-20px for source/markdown, 0 for image (edge-to-edge).
25. Message Bubbles
Chat messages use asymmetric styling. User messages right-align with a surface bubble, max-width 80%. Assistant messages left-align with no bubble background — they take full width on canvas, letting content (paragraphs, code blocks, tool calls, file outputs) flow naturally without nesting visual containers.

A note on inline SVG icons
A handful of inline-SVG icon components still exist in the codebase (TrashIcon, SendIcon, PaperclipIcon, EyeIcon, ThemeIcon, etc.) — holdovers from before the Bootstrap Icons migration. The canonical library going forward is Bootstrap Icons. Inline SVGs are being migrated to their bi-* equivalents; any new UI should use the Bootstrap class directly rather than introducing another inline SVG.
